# YARG Venue Research

This project should model venue data around YARG's chart-driven venue system rather than folding it into timeline or preview-only state.

## YARG venue track shape

YARG consumes a `VenueTrack` with these top-level buckets:

- `Lighting`: chart-authored lighting cues, plus `KeyframeNext` sequences for manual patterns.
- `PostProcessing`: chart-authored post-processing events.
- `Stage`: stage effects such as `BonusFx`, `FogOn`, and `FogOff`.
- `Performer`: performer-directed events such as spotlights and singalong-style cues.
- `CameraCuts`: authored camera cut subjects.
- `AutoGenerated`: flag indicating that venue events were generated rather than authored.

Relevant YARG runtime anchors found during research:

- `Assets/Script/Integration/MasterLightingGameplayMonitor.cs`
- `Assets/Script/Venue/Lights/LightManager.cs`

## Important YARG behavior

- If `Chart.VenueTrack.IsEmpty`, YARG first tries to load venue data from milo.
- If fog is missing, YARG may auto-generate fog events even when other venue data exists.
- If lighting is missing, YARG may auto-generate lighting and post-processing from section names using `DefaultVenuePreset.json`.
- Lighting, post-processing, and stage events are treated as separate ordered tracks.
- Camera cuts and performer events exist in the model, but comments in `MasterLightingGameplayMonitor` indicate some parser/runtime paths are still NYI there.
- Post-processing compatibility matters because YARG uses many named effect types, not just a generic effect flag.
- Stage events currently include at least `BonusFx`, `FogOn`, and `FogOff`.
- Venue backgrounds in YARG are separate from chart venue events. They can be song/global/default image, video, or `.yarground` assets.

## What this means for chart-editor

- `videoSync` and `audioSync` are not the venue system.
- Venue authoring should have its own `venueTrack` in song data.
- We should preserve raw YARG event names rather than collapsing them early.
- Parser work should target YARG-compatible event buckets first, even before the UI is polished.
- Venue background assets can be handled later; the first compatibility target is chart venue event data.

## Recommended implementation order

1. Parse and serialize `venueTrack` alongside existing note/chart data.
2. Start with `lighting`, `postProcessing`, and `stage` because they have the clearest runtime behavior in YARG.
3. Add a simple timeline/editor lane for those venue events.
4. Add `cameraCuts` and `performer` editing after the base parser round-trip is stable.
5. Consider milo import and auto-generation only after authored event round-trip works.
